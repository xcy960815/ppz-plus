# 工作流约束

更新时间：2026-06-30

本文档记录日常实现、验证和协作时的工作规则。这里放操作性规则，项目背景和架构原因分别见 `project-memory.md` 与 `architecture.md`。

## 包管理与脚本

- 项目强制使用 `pnpm` 作为包管理器。
- 安装依赖、执行脚本、编译或测试时都必须使用 `pnpm`。
- 不要使用 `npm` / `yarn`。

## 构建与验证

- 修改功能或修复 bug 之后，默认不要主动跑构建。
- 是否执行构建、打包、完整验证，由后续明确需要时再单独决定。
- 如果用户明确要求验证，优先使用项目已有脚本，例如 `pnpm compile`、`pnpm lint`、`pnpm test`。

## 修复策略

- 修 bug 时只修 ppz-plus 当前新架构中的真实问题。
- 不为了旧 PPZ 的历史实现、旧返回结构或旧兼容路径额外增加兼容分支。
- 遇到字段名、返回结构或历史行为不一致时，必须通过当前新架构的明确模型、SQL alias、类型定义或数据契约修正。
- 不允许写 `a ?? b ?? legacyC` 这类多形态兼容兜底。

## 代码注释

- 新增或修改代码时，变量和方法默认补充标准 JSDoc 注释。
- 注释描述文本使用中文。
- 只在能帮助理解边界、意图或复杂逻辑时写注释，避免机械重复代码字面含义。

## Todo 维护

- 功能迁移和能力支持进度统一维护在 `.agents/todolist.md`。
- 任意一项真正完成后，在对应条目后追加 `✅`。
- 未明确完成前不要提前打勾。
- 新增功能时，先补充 todo，再开始实现。

## Changelog

- Changelog 自动化规则见 `.agents/changelog-automation.md`。
- 日常开发保持 commit message 符合 Conventional Commits 规范，便于后续生成 changelog。
