# PPZ 旧项目能力审计与 ppz-plus 重写基线

更新时间：2026-06-27

本文档用于作为 `ppz-plus` 的长期项目记忆。目标不是复用旧项目 `PPZ` 的实现，而是把旧项目视为功能需求来源和历史产品原型，为新项目提供明确的迁移边界。

## 结论摘要

- `ppz-plus` 继承的是“产品能力”，不是旧代码实现。
- 旧项目中值得保留的是：连接管理、Tree View、表数据浏览、SQL 终端、导出能力、多数据库方向。
- 旧项目中应整体重写的是：命令系统、Tree 内部模型、连接服务、Webview 基建、导入导出架构、状态与迁移体系、测试与任务体系。
- 第一阶段应以 MySQL 为唯一一等公民数据库，先打通可维护的新架构闭环。

## 研究范围

本结论同时参考了旧仓库两个基线：

- 旧仓库当前 `main`
- 历史发布标签 `v0.5.1-beta`

其中：

- 当前 `main` 更像“重构中的骨架”，不能代表旧产品完整能力。
- `v0.5.1-beta` 更接近历史发布版真实能力，因此本文大部分产品能力判断以该标签为准。

## 判断标记规则

- `代码事实`：直接来自旧仓库代码、目录、命令清单、适配器实现、测试材料。
- `推断/建议`：基于代码现状得出的架构判断、新项目实施建议。

## 1. 旧 PPZ 能力拆解清单

### 1.1 插件入口与基础结构

- `代码事实`：`v0.5.1-beta/package.json` 是完整 VS Code 扩展清单，注册了视图容器、Tree View、命令、右键菜单。
- `代码事实`：旧仓库当前 `main/package.json` 仅表现为构建入口，不是完整插件产品清单。
- `代码事实`：`v0.5.1-beta/app/extension.js` 启动时会初始化命令、状态迁移、依赖复制、Tree View。

判断：

- `推断/建议`：旧仓库当前 `main` 不应作为迁移基线。

### 1.2 命令系统

- `代码事实`：`v0.5.1-beta/package.json` 注册了以下主要命令：
  - `ppz.addConnection`
  - `ppz.editConnection`
  - `ppz.deleteConnection`
  - `ppz.reloadTreeItemChildren`
  - `ppz.ppzTerminal`
  - `ppz.exportDML`
  - `ppz.exportDDL`
  - `ppz.exportBoth`
- `代码事实`：还有帮助类命令，如 `ppz.github`、`ppz.empty`。
- `代码事实`：`app/command/connection.js` 里命令会直接操作 Tree 元素、直接打开 Webview、直接调用连接对象导出。

判断：

- `推断/建议`：用户可见命令能力有保留价值，但命令层职责混乱，必须重写。

### 1.3 Tree View 能力

- `代码事实`：左侧 Activity Bar 中存在独立容器和连接视图。
- `代码事实`：不同数据库的树层级如下：
  - MySQL：连接 -> database/schema -> table
  - PostgreSQL：连接 -> database -> schema -> table
  - MSSQL：连接 -> database -> schema -> table
  - SQLite3：连接 -> table
- `代码事实`：Tree 节点上挂接了刷新、编辑、删除、导出、SQL 终端、打开表等操作。
- `代码事实`：不同类型节点的右键菜单不同，菜单按数据库差异做了分流。

判断：

- `推断/建议`：Tree 的用户交互模型值得继承。
- `推断/建议`：Tree 节点内部直接创建/缓存数据库连接的做法不应继承。

### 1.4 连接管理

- `代码事实`：存在“新增连接”和“编辑连接” Webview。
- `代码事实`：连接配置支持：
  - MySQL
  - PostgreSQL
  - MSSQL
  - SQLite3
  - CockroachDB
- `代码事实`：连接配置支持两种输入模式：
  - 参数模式
  - URL 模式（部分数据库）
- `代码事实`：SQLite3 使用文件选择器选数据库文件。
- `代码事实`：连接删除能力已实现。

判断：

- `推断/建议`：连接管理能力应直接迁移。
- `推断/建议`：旧连接模型与持久化格式不应作为新架构边界。

### 1.5 表数据浏览与数据管理

- `代码事实`：表页面支持：
  - 分页
  - 排序
  - 搜索
  - 字段显隐控制
  - 查看当前 SQL
  - 复制 SQL
  - 刷新
  - 新增记录
  - 更新记录
  - 删除记录
  - 打开 SQL 终端
- `代码事实`：无主键表会限制编辑和删除操作。
- `代码事实`：表页面通过 Webview 与后端通信，调用 `fieldList`、`select`、`updateMany`、`drop`、`insert`。

判断：

- `推断/建议`：表数据浏览是旧产品的核心价值，必须保留。
- `推断/建议`：行级写操作是否进入第一阶段，需要看架构落地节奏，不能为了追平旧版而过早接入。

### 1.6 SQL 终端

- `代码事实`：存在专门的 SQL 终端 Webview。
- `代码事实`：终端本质上直接执行 `connection.client.raw(sql)`。
- `代码事实`：前端根据不同驱动显示不同结果视图。

判断：

- `推断/建议`：SQL 终端能力值得保留。
- `推断/建议`：执行层、结果适配层、UI 层要拆开，不能复用旧版“直接 raw + 直接展示”的结构。

### 1.7 导出能力

- `代码事实`：支持三个用户入口：
  - 导出 DML
  - 导出 DDL
  - 导出 DDL + DML
- `代码事实`：支持从 schema 节点和表节点发起导出。
- `代码事实`：导出结果是打开 untitled SQL 文本，不是导出任务或落盘工作流。
- `代码事实`：各数据库实现并不一致：
  - MySQL：DDL 可用，DML 可用
  - SQLite3：DDL 可用，DML 可用
  - PostgreSQL：DML 可用，DDL 明确不支持
  - MSSQL：DML 可用，DDL 明确不支持
  - CockroachDB：基于 PostgreSQL 适配，DDL 能力也不完整

判断：

- `推断/建议`：导出能力本身值得继承。
- `推断/建议`：旧版导出实现只是“能用”，不是可扩展架构，必须重写。

### 1.8 导入能力

- `代码事实`：未发现导入命令、导入服务、导入任务、导入解析器。
- `代码事实`：当前只有“新建一条记录”的插入能力，不能视为导入。

判断：

- `推断/建议`：旧版基本没有真正的导入能力。
- `推断/建议`：`ppz-plus` 的导入能力应按新需求独立设计，不能以旧版为基础。

### 1.9 多数据库支持现状

- `代码事实`：旧版真实存在的数据库适配器为：
  - MySQL
  - PostgreSQL
  - SQLite3
  - MSSQL
  - CockroachDB
- `代码事实`：README 中提到 MariaDB、StoneDB 等，但代码里没有独立适配器实现。

判断：

- `推断/建议`：MariaDB、StoneDB 更可能是协议兼容带来的“可连”，不是一等公民支持。

### 1.10 测试现状

- `代码事实`：`v0.5.1-beta` 几乎没有自动化测试，只有少量 SQL mock 文件。
- `代码事实`：`FCT.md` 明确以手工功能测试为主。

判断：

- `推断/建议`：测试体系需要从头建立。

## 2. 哪些能力应该直接迁移，哪些能力应该延后，哪些能力应该放弃

### 2.1 直接迁移的能力

- 连接新增、编辑、删除、持久化
- 左侧数据库资源 Tree View
- 表数据只读浏览
- SQL 执行/查询终端
- DDL/DML 导出入口
- 多数据库分阶段演进方向
- 能力差异提示机制

### 2.2 建议延后的能力

- 表数据写入：新增、更新、删除
- 真正的导入能力
- 跨库/跨 schema 批量导出任务
- Webview 状态恢复细节
- SQLite3 支持
- MSSQL 支持
- CockroachDB 支持

### 2.3 建议放弃的能力或实现方式

- 旧版“系统终端”思路
- SQLite3 依赖手工复制/提示安装模式
- 旧式“一条命令直接碰 UI + 数据库 + 通知”的写法
- 旧版按连接对象隐式暴露所有能力的“大而全连接类”

## 3. 哪些模块适合参考旧实现，哪些模块建议完全重写

### 3.1 可参考旧实现的部分

- 各数据库元数据查询的大致方向
- Tree 层级设计
- 连接表单的字段矩阵
- 表数据页交互能力列表
- 日期/时间类型处理里暴露过的问题点
- 导出入口应该出现在哪些节点上

### 3.2 建议完全重写的部分

- 扩展入口与 bootstrap
- 命令注册与命令分发
- Tree 节点内部模型
- 连接生命周期管理
- Webview 通信层
- 状态存储与版本迁移
- 导入导出架构
- 数据库适配器接口
- 长任务/后台任务体系
- 自动化测试与测试夹具

### 3.3 为什么建议重写

- `代码事实`：旧版 `service/connection/index.js` 会直接更新 Tree View。
- `代码事实`：旧版 `KnexConnection` 同时承担查询、格式化、导出、打开终端、通知、打开 untitled 文件等职责。
- `代码事实`：Tree 节点会惰性创建连接，并在不同层级重复建连。
- `代码事实`：DDL 能力按数据库散落在适配器内部，没有统一能力声明。

判断：

- `推断/建议`：旧实现已经形成高耦合结构，不适合作为长期维护基础。

## 4. 新项目推荐架构分层

建议采用如下分层：

### 4.1 Presentation 层

负责：

- VS Code commands
- Tree View provider
- Webview panels
- 通知、错误展示、进度条、QuickPick 等

原则：

- 不直接写数据库逻辑
- 不直接拼 SQL
- 不直接做持久化

### 4.2 Application 层

负责：

- 用例编排
- 参数校验
- 多服务协调
- 任务发起与取消

示例用例：

- 创建连接
- 测试连接
- 加载树节点
- 打开表数据
- 执行 SQL
- 导出 DDL
- 导出 DML
- 导入数据

### 4.3 Domain 层

负责：

- 连接配置模型
- 资源树模型
- 数据库能力模型
- 查询结果模型
- 导入导出规范
- 错误模型

原则：

- 不依赖 `vscode`
- 不依赖具体数据库驱动

### 4.4 Infrastructure 层

负责：

- VS Code API 适配
- 全局状态/SecretStorage/文件存储
- 数据库驱动封装
- 文件系统
- 任务执行器

## 6. 各系统职责划分建议

### 6.1 命令系统

应负责：

- 用户入口
- 参数收集
- 调用 use case

不应负责：

- 直接操作连接对象
- 直接刷 Tree
- 直接执行 SQL

### 6.2 Tree View

应负责：

- 展示资源树
- 根据节点类型绑定命令
- 拉取树节点 DTO

不应负责：

- 持有数据库连接
- 直接发起导出
- 维护业务状态

### 6.3 Webview

应负责：

- 用户交互
- 展示查询结果
- 发送结构化请求

不应负责：

- 自己定义数据库能力规则
- 直接决定驱动行为

### 6.4 任务系统

建议独立出来，负责：

- 导入/导出
- 长时间查询
- 进度汇报
- 取消
- 错误日志

### 6.5 数据库适配层

不要继续使用单一“大连接类”。

建议拆成能力接口：

- `ConnectionTester`
- `MetadataProvider`
- `QueryExecutor`
- `DataPreviewProvider`
- `ExportProvider`
- `ImportProvider`

每个数据库按能力实现，不支持的能力明确声明。

### 6.6 导入导出层

应是独立子系统，而不是适配器里的若干零散方法。

建议职责：

- 导出目标解析
- 导出格式选择
- DDL/DML 生成
- 结果写入策略
- 导入源解析
- 导入执行任务
- 失败恢复与日志

## 7. 多数据库支持如何按阶段推进

### 阶段 1

- 仅 MySQL
- 打通完整闭环

### 阶段 2

- PostgreSQL
- 重点验证 database/schema 双层模型
- 补齐能力声明差异

### 阶段 3

- SQLite3
- 前提是先设计清楚驱动分发与安装策略

### 阶段 4

- MSSQL

### 阶段 5

- CockroachDB / MariaDB / 其他兼容型数据库
- 按 capability profile 接入

## 8. 第一阶段最小可用范围

建议第一阶段 MVP 只做：

- MySQL 连接管理
- 测试连接
- Tree View
- schema/table 浏览
- 表数据只读浏览
- 分页、排序、过滤
- SQL 执行
- 表级 DDL/DML 导出

第一阶段不建议包含：

- 批量导入
- 行级写操作
- SQLite3
- MSSQL
- CockroachDB
- 跨数据库统一导出任务

## 9. 最容易因为“试图兼容旧代码”而把新项目拖死的地方

- 复用旧 Tree Element 继承体系
- 复用旧连接服务
- 复用旧 Webview 请求基建
- 把旧连接存储格式当成新领域模型
- 一开始就追求所有数据库同时可用
- 把“数据库支持”误认为“所有数据库同等能力”
- 沿用旧版“导出就是打开 untitled SQL 文件”的唯一模式

## 10. 明确结论：哪些按“继承能力”处理，哪些按“推倒重来”处理

### 10.1 按“继承能力”处理

- 连接管理
- Tree View 浏览
- 表数据查看
- SQL 终端
- 导出入口
- 多数据库产品方向

### 10.2 按“推倒重来”处理

- 插件工程结构
- 架构分层
- 命令系统
- Tree 内部模型
- 连接生命周期
- Webview 基建
- 导入导出子系统
- 多数据库适配接口
- 任务系统
- 状态与迁移体系
- 自动化测试体系

## 附：本次研究的关键代码事实

- 旧仓库当前 `main` 仅保留很薄的重构骨架。
- `v0.5.1-beta` 才是真实产品能力的主要依据。
- 旧版确有连接管理、Tree View、表数据页、SQL 终端、导出。
- 旧版没有真正的导入体系。
- PostgreSQL 与 MSSQL 的 DDL 导出明确不完整。
- 自动化测试基本缺失。

## 对 ppz-plus 的直接行动约束

- 不以旧仓库 `main` 为迁移基线。
- 不以“兼容旧代码”为目标。
- 以 “MySQL MVP -> PostgreSQL -> 其他数据库” 的顺序推进。
- 先建清晰架构，再接具体数据库能力。
